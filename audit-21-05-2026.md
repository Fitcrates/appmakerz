# Audyt Wydajności i Konfiguracji (Next.js + Netlify)

Przeprowadziłem dogłębną analizę przekazanych plików pod kątem routingu, tranzycji, cachingu i specyfiki hostingu na Netlify. Znalazłem kilka kluczowych problemów (w tym jeden **krytyczny błąd konfiguracyjny**), które bezpośrednio wpływają na wydajność, szybkość działania animacji (stuttering) oraz opóźnienia w nawigacji.

Poniżej zestawienie wyników audytu:

## 1. KRYTYCZNE: Błędny Caching Skryptów Next.js w `netlify.toml`

W Twoim pliku `netlify.toml` zdefiniowano następujące nagłówki:

```toml
[[headers]]
  for = "/*"
  [headers.values]
    Cache-Control = "public, max-age=0, must-revalidate"
    # ...
```

Później nadpisujesz to dla statycznych folderów: `/assets/*`, `/media/*`, `/fonts/*`, `/static/*` (ustawiając `max-age=31536000, immutable`). 

**Gdzie leży problem?**
Next.js kompiluje i serwuje wszystkie swoje pliki JavaScript oraz CSS do ścieżki `/_next/static/*`. Twój plik konfiguracyjny nie uwzględnia tej ścieżki, co oznacza, że łapie się ona na ogólną regułę `/*` (`max-age=0, must-revalidate`).
W efekcie przeglądarka **przy każdej nawigacji** musi pytać serwer (Netlify), czy pliki JS/CSS się zmieniły. Zamiast wczytywać przejścia i komponenty z pamięci podręcznej w 0ms, przeglądarka czeka na odpowiedź sieciową. To bezpośrednio powoduje **przycięcia, gubienie klatek w animacjach i opóźnienia w zmianie routingu**.

**Rozwiązanie:**
Dodaj regułę dla katalogu `_next/static` w `netlify.toml`:

```toml
[[headers]]
  for = "/_next/static/*"
  [headers.values]
    Cache-Control = "public, max-age=31536000, immutable"
```

## 2. WYDAJNOŚĆ LCP: Podwójna Optymalizacja Obrazków (Sanity + Next.js)

W plikach takich jak `[slug]/page.tsx` czy `uslugi/[slug]/page.tsx` używasz poniższego wzorca:

```tsx
const heroImageUrl = project.mainImage ? urlFor(project.mainImage).width(1200).height(630).auto('format').fit('crop').url() : '';

// ... i później:
<Image src={heroImageUrl} alt={title} fill priority sizes="(max-width: 768px) 100vw, 1200px" />
```

**Gdzie leży problem?**
Twój `heroImageUrl` jest już w pełni zoptymalizowanym obrazkiem zwracanym przez potężny CDN Sanity (ma ustawiony wymiar i auto-format, np. na WebP). Przekazując ten adres do komponentu `<Image>` z Next.js, zmuszasz serwer/Edge Netlify do pobrania tego obrazka z Sanity, zoptymalizowania go **ponownie** za pomocą wbudowanego silnika Next.js i dopiero zaserwowania użytkownikowi. To opóźnia LCP (Largest Contentful Paint) i niepotrzebnie obciąża limit optymalizacji na Netlify.

**Rozwiązanie:**
Jeśli url jest już optymalizowany przez Sanity, wyłącz optymalizację w Next.js dodając flagę `unoptimized` do kluczowych, dużych obrazów:
```tsx
<Image src={heroImageUrl} unoptimized fill priority ... />
```
Alternatywnie, skorzystaj ze zwykłego tagu `<img>`, lub zaimplementuj własny `loader` dla komponentu `Image`, który bezpośrednio mapuje parametry na URL Sanity bez pośrednictwa backendu Next.js.

## 3. ISR i Caching Stron

Wszystkie Twoje strony używają bardzo dobrego wzorca ISR (Incremental Static Regeneration):
```tsx
export const revalidate = 3600;
export const dynamic = 'force-static';
```
Dla stron dynamicznych (np. `/blog/[slug]`) poprawnie używasz `export const dynamicParams = true;`.

**Ocena:** To świetne rozwiązanie. Netlify wspiera to natywnie. Oznacza to, że strony ładują się jak czysty HTML z brzegowego CDN (Edge). Tu nie ma żadnego problemu z wydajnością, pod warunkiem, że punkt 1 zostanie naprawiony.

## 4. Architektura Tranzycji (RouteTransitionProvider)

Jak ustaliliśmy wcześniej, rozdzielenie `NAVIGATE_DELAY_MS` i `COVER_DURATION_MS` naprawiło błąd z zawieszaniem się tła.
Warto jednak pamiętać, że jeśli `NAVIGATE_DELAY_MS` jest krótkie (np. 200ms), to Next.js bardzo wcześnie podmienia strukturę DOM (montuje nową stronę). Animacja Canvas ukrywa to przed oczami użytkownika.

**Wskazówka:** Jeśli nowa strona (np. `project/[slug]`) posiada ciężkie animacje na wejście (np. framer-motion, które odpalają się natychmiast po zamontowaniu), to mogą one "przelecieć" zanim opadnie Canvas w fazie `reveal`. Warto powiązać animacje wejściowe komponentów z globalnym stanem z `RouteTransitionContext`, by zaczynały się dopiero po zakończeniu fazy `cover`.

## 5. Pozostałe detale (netlify.toml)

W Twoim pliku `netlify.toml` widnieje zapis:
```toml
[[plugins]]
  package = "@netlify/plugin-nextjs"
```
Od wersji Next.js Runtime v5 na Netlify, ten plugin jest preinstalowany domyślnie i automatycznie integrowany. Pozostawienie go w pliku konfiguracyjnym zazwyczaj nie psuje strony, ale jest traktowane jako "legacy" i w niektórych scenariuszach może powodować użycie starszej warstwy kompatybilności. Zalecam usunięcie tego bloku, pozwalając Netlify na automatyczne zastosowanie natywnego środowiska wykonawczego dla Next.js.

### Podsumowanie: Co musisz wdrożyć?

1. **Natychmiast** dodaj regułę buforowania dla `/_next/static/*` w `netlify.toml`. To absolutnie kluczowe dla zlikwidowania przycięć podczas przechodzenia po zakładkach.
2. Dodaj flagę `unoptimized` w komponencie `next/image` dla zdjęć, które przechodzą już przez `urlFor()` Sanity.
3. Usuń przestarzały blok `[[plugins]]` z `netlify.toml`.
